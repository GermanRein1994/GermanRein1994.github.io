Descripción del Caso Práctico

El proyecto consiste en un workflow automatizado que opera un juego de trivia a través de Telegram. Los usuarios interactúan con el bot para responder preguntas, gestionar un sistema de vidas y acumular puntaje según distintos niveles de dificultad. El estado de las partidas y el inventario de preguntas se administran de forma persistente utilizando Google Sheets.

Para manejar la imprevisibilidad de los usuarios (quienes pueden escribir texto libre en lugar de usar los botones de la interfaz), el flujo integra inteligencia artificial para orquestar y dinamizar la interacción siguiendo este ciclo:

  1. Trigger Automático: el flujo se activa mediante un Telegram Trigger que escucha en tiempo real cualquier mensaje de texto o pulsación de botón en el chat del bot.
  2. LLM de Clasificación (Orquestador): cuando se detecta texto libre, la entrada se envía a un primer modelo de lenguaje (Llama 3 vía Groq) configurado estrictamente para tareas de clasificación semántica. El LLM deduce la intención del usuario y devuelve únicamente un objeto JSON estructurado con la llave "accion" (pudiendo tomar los valores: responder, jugar, reglas o reprender).
  3. Lógica Condicional: un nodo tipo Switch actúa como enrutador leyendo la salida estructurada del LLM. Si la acción es inválida o es una conversación fuera de contexto, el flujo se deriva por la rama "reprender".
  4. LLM de Generación de Contenido (Interacción): la rama "reprender" alimenta a un segundo modelo de lenguaje con dos variables: el mensaje irrelevante del usuario y el contexto actual de la base de datos (si tiene una partida activa o no). Este LLM asume el rol de "anfitrión sarcástico", redactando una respuesta dinámica para encarrilar al usuario.
  5. Notificaciones y Métricas: las salidas (ya sean del motor del juego o las respuestas redactadas por el LLM) se envían de vuelta al usuario como mensajes de Telegram, modificando la interfaz para ocultar botoneras obsoletas. Adicionalmente, se registra un log de las respuestas en una hoja de Histórico para métricas y posterior análisis.

Demostración del Bot en Acción

Módulo de Ingesta, Unificación y Notificación de Fallas

  • Recibe el Trigger de Telegram y ejecuta lecturas paralelas a Google Sheets.
  • El nodo Unificar_datos normaliza los IDs, cruza la información de sesión y empaqueta todo en un único objeto JSON que viaja por el resto del flujo.
  • El módulo de notificación de fallas envía un mensaje al usuario si hubo un problema en la lectura y/o actualización de los datos de las hojas de cálculo.

Módulo de Gestión de Usuarios y Control de Sesión

  • Verifica si el usuario existe en la base de datos. Si no, lo crea.
  • Control 24hs: Evalúa el timestamp de la ultima_interaccion. Si supera las 24 horas de inactividad, limpia las variables de estado y le notifica al usuario que su partida caducó.

Módulo de Orquestación con LLM

  • Los mensajes de texto que no son comandos directos entran al nodo condicional de texto.
  • Orquestador_LLM: Recibe las opciones de la pregunta actual y el texto del usuario. Devuelve una instrucción estricta de enrutamiento.
  • Switch Semántico: Deriva a asignar la respuesta sintética, a enviar las reglas del juego, o al Módulo de Interacción.
  • LLM_Interaccion: Genera una respuesta sarcástica requiriendo que el usuario vuelva al flujo de la trivia si el mensaje original era irrelevante.

Módulo de Lógica de Juego (Setup y Evaluación)

  • Preparar Partida: Configura vidas (5, 3 o 1) y multiplicadores de puntaje (x1, x1.5, x2) según la dificultad seleccionada. Realiza el muestreo aleatorio de preguntas por categoría.
  • Evaluar Respuesta: Núcleo matemático del juego. Compara resultados, suma puntos aplicando el multiplicador, resta vidas y actualiza el récord.
  • Limpieza de Interfaz: Nodos dedicados que utilizan las APIs de Telegram para eliminar los teclados en línea (Inline Keyboards) de los mensajes anteriores, evitando saturar el chat.
  • Actualizar Histórico: Por cada respuesta evaluada, se inserta una nueva fila en una hoja de cálculo registrando parámetros clave (timestamp, dificultad, ID de pregunta, resultado) para su posterior ingesta en herramientas de Business Intelligence.
  • Insistencia y Reglas: Si el usuario rechaza jugar, se calcula un nivel de rechazo para escalar la presión en el mensaje devuelto. Si pide las reglas, se verifica la ruta de origen para determinar si hace falta borrar botones previos.

Buenas Prácticas Aplicadas

  • Prevención "Anti-Zombie" (Idempotencia): Verificación estricta que cruza el ID del botón presionado contra el ID de la base de datos, previniendo fallos por doble contabilización si un usuario presiona botones viejos.
  • Tolerancia a Fallos (Error Handling): Nodos configurados con políticas de reintento y continuación en caso de caídas de APIs externas, junto con notificaciones automáticas de mantenimiento al usuario.
  • Filtrado de Carga Útil: Validación nativa para ignorar envíos de imágenes, audios o stickers, asegurando que los nodos de procesamiento LLM solo operen con texto válido.
